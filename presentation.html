Agent architecture

Coordinator ← claim_id / data/inbox/ │ ├─ Task → DocumentReader (isolated context) │          fetch_claim · parse_attachments │          → ClaimSummary │ ├─ Task → PolicyChecker (receives ClaimSummary only) │          lookup_policy · check_fraud_flags · check_sanctions │          → PolicyResult │ ├─ validate + retry (max 3 · schema error fed back) ├─ escalation rules (€5K · conf<0.75 · fraud · sanctions) └─ write_decision() or escalate_claim()

Task subagents receive explicit context strings — they do not inherit coordinator memory. This is not optional: omitting it causes silent hallucination of missing fields.

Key architectural decisions — serialised automatically

Thresholds are explicit numbers (€5 000, confidence 0.75) rather than vague guidance ("when the agent isn't sure"). Explicit thresholds produce consistent, auditable escalation behavior that can be reviewed by IVASS and Legal. They also make the eval harness deterministic: the scorecard can distinguish correct escalations from needless ones because the rule is unambiguous.

The coordinator + specialist split was chosen over a single monolithic agent because the two knowledge domains (document parsing vs. policy reasoning) have different tool sets and different failure modes. Keeping them separate makes each one easier to test, easier to reason about, and easier to replace if the policy rules change.

"Writing CLAUDE.md first — with € not $, PII rules, threshold ownership — meant every file was already oriented correctly before implementation. The conventions propagated themselves."
