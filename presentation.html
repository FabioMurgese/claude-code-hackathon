Challenge 2 & 4 — The Bones + The Triage

Coordinator + Specialist architecture

Coordinator ← claim_id from data/inbox/ │ ├─ Task → DocumentReader (isolated context) │          fetch_claim · parse_attachments │          → ClaimSummary │ ├─ Task → PolicyChecker (receives ClaimSummary only) │          lookup_policy · check_fraud_flags · check_sanctions │          → PolicyResult │ ├─ validate + retry loop (max 3 · schema error fed back) ├─ escalation rules (€5 000 · conf < 0.75 · fraud · sanctions) │ └─ write_decision() OR escalate_claim()

Task subagents do not inherit coordinator context. Every specialist receives an explicit context string in its Task prompt — ClaimSummary for PolicyChecker, claim_id for DocumentReader.

Challenge 5 — The Brake

Hard stops + slow stops.
Two layers, by design.

The hook is deterministic — no LLM involved. The escalation rules are probabilistic. Both are required.

PreToolUse hook — hard stops

Regex patterns run before any tool call. Cannot be bypassed by prompt injection.

Pattern detected Block code
Codice Fiscale in any tool input GDPR_PII_BLOCKED
Partita IVA / IBAN IT pattern GDPR_PII_BLOCKED
External URL in tool input EXTERNAL_ROUTING_BLOCKED
write_decision on frozen polizza FROZEN_ACCOUNT_BLOCKED
approve + fraud_score > 0 FRAUD_APPROVE_BLOCKED

Escalation rules — slow stops

Explicit thresholds, not vague guidance. Auditable by IVASS and Legal.

Trigger Threshold
Euro amount ≥ €5 000
Decision confidence < 0.75
Fraud score (D.Lgs. 231) > 0
Sanctions hit (EU/UN) true
Coverage status ambiguous
Claim type contestazione

Explicit thresholds produce consistent, auditable escalation behavior. Vague rules ("when the agent isn't sure") produce inconsistent rates that cannot be defended in a regulatory review.

Architectural key decisions — extracted automatically from docs

> Key Decision — Mandate
Thresholds are explicit numbers (€5 000, confidence 0.75) rather than vague guidance ("when the agent isn't sure"). Explicit thresholds produce consistent, auditable escalation behavior that can be reviewed by IVASS and Legal. They also make the eval harness deterministic: the scorecard can distinguish correct escalations from needless ones because the rule is unambiguous.
> Key Decision — Architecture
The coordinator + specialist split was chosen over a single monolithic agent because the two knowledge domains (document parsing vs. policy reasoning) have different tool sets and different failure modes. Keeping them separate makes each one easier to test, easier to reason about, and easier to replace if the policy rules change.
