Agent Architecture

Coordinator ← claim_id from data/inbox/
│
├─ Task → DocumentReader (isolated context)
│         fetch_claim · parse_attachments
│         → ClaimSummary
│
├─ Task → PolicyChecker (receives ClaimSummary only)
│         lookup_policy · check_fraud_flags · check_sanctions
│         → PolicyResult
│
├─ validate + retry loop (max 3, schema error fed back)
├─ escalation rules (€5 000 · conf < 0.75 · fraud · sanctions)
└─ write_decision() OR escalate_claim()

Italian/EU Market Adjustments

Codice Fiscale PII blocking Partita IVA blocking IBAN IT pattern € thresholds D.Lgs. 231/2001 fraud rules EU/UN sanctions list IVASS regulation GDPR guardrails

Decisions the agent makes

fast_track < €5 000, clear coverage, no flags
investigate borderline amount or ambiguous coverage
deny lapsed policy, exclusion, duplicate
escalate ≥ €5 000, fraud, sanctions, contestazione

Key architectural decisions — surfaced automatically

🔑 Key Decision — Mandate
Thresholds are explicit numbers (€5 000, confidence 0.75) rather than vague guidance ("when the agent isn't sure"). Explicit thresholds produce consistent, auditable escalation behavior that can be reviewed by IVASS and Legal. They also make the eval harness deterministic: the scorecard can distinguish correct escalations from needless ones because the rule is unambiguous.
🔑 Key Decision — Architecture
The coordinator + specialist split was chosen over a single monolithic agent because the two knowledge domains (document parsing vs. policy reasoning) have different tool sets and different failure modes. Keeping them separate makes each one easier to test, easier to reason about, and easier to replace if the policy rules change.
